Common Criteria for IT Security Evaluation

What is ISO/IEC 15408 (Common Criteria)?

ISO/IEC 15408, commonly known as Common Criteria (CC), is an international standard for computer security certification. It provides a framework for specifying security functional and assurance requirements for IT products and systems. Common Criteria serves as a basis for evaluating the security properties of these products through a standardized approach that is recognized globally.

Key Facts About ISO/IEC 15408

  • Governing Body: International Organization for Standardization (ISO) and International Electrotechnical Commission (IEC)
  • Primary Region: Global (recognized by 31 countries under the Common Criteria Recognition Arrangement)
  • Legal Authority: Voluntary standard, but often required for government procurement
  • Compliance Requirement: Product-specific certification through formal evaluation
  • Focus Areas: IT product security, security assurance, protection profiles

Common Criteria Recognition Arrangement (CCRA)

The Common Criteria Recognition Arrangement (CCRA) is an international agreement that establishes a framework for mutual recognition of Common Criteria certifications among participating countries. This means that a product certified in one CCRA member country is recognized as certified in all other member countries, eliminating the need for multiple evaluations.

Current CCRA member countries include:

  • Australia, Austria, Canada, Czech Republic, Denmark, Finland, France, Germany, Greece, Hungary, India, Israel, Italy, Japan, Malaysia, Netherlands, New Zealand, Norway, Pakistan, Qatar, Republic of Korea, Singapore, Spain, Sweden, Turkey, United Kingdom, United States

Common Criteria Structure

The Common Criteria standard consists of three parts:

Part Title Description
Part 1 Introduction and General Model Provides an overview of the CC, defines general concepts and principles, and explains how to select appropriate security requirements
Part 2 Security Functional Requirements Establishes a set of functional components as a standard way of expressing security requirements for IT products
Part 3 Security Assurance Requirements Establishes a set of assurance components as a standard way of expressing assurance requirements for IT products

Key Concepts in Common Criteria

Protection Profile (PP)

A Protection Profile is an implementation-independent set of security requirements for a category of products that meet specific consumer needs. PPs are created to allow consumers or communities of interest to express their security needs without being tied to specific implementations.

Security Target (ST)

A Security Target is a document that identifies the security properties of a specific product. It includes the security functional and assurance requirements that the product claims to meet, and it forms the basis for evaluation.

Target of Evaluation (TOE)

The Target of Evaluation is the actual product or system being evaluated. The TOE can be a complete product, a part of a product, or a set of products.

Evaluation Assurance Level (EAL)

Evaluation Assurance Levels define the depth and rigor of an evaluation. There are seven levels, with EAL1 being the most basic and EAL7 being the most stringent:

Level Description Typical Applications
EAL1 Functionally Tested Products where some confidence in correct operation is required but threats are not serious
EAL2 Structurally Tested Products requiring low to moderate level of independently assured security
EAL3 Methodically Tested and Checked Products requiring moderate level of independently assured security with thorough investigation
EAL4 Methodically Designed, Tested, and Reviewed Products requiring moderate to high level of independently assured security for commercial products
EAL5 Semiformally Designed and Tested Products requiring high level of independently assured security in a planned development environment
EAL6 Semiformally Verified Design and Tested Products for high-risk situations where the value of protected assets justifies additional cost
EAL7 Formally Verified Design and Tested Products for extremely high-risk situations and high-value assets

Note on EAL Levels

Under the current CCRA, mutual recognition is limited to certifications up to EAL2 (and EAL4 for certain Protection Profiles). Higher EAL certifications may be issued by individual countries but are not automatically recognized by other CCRA members.

Security Functional Requirements (SFRs)

Common Criteria Part 2 defines a comprehensive set of security functional requirements organized into classes, families, and components. These SFRs are used to express the security functionality that a product must provide.

Class Description Example Families
FAU: Security Audit Requirements for recognizing, recording, storing, and analyzing security-relevant activities FAU_GEN (Audit data generation), FAU_SAR (Audit review)
FCO: Communication Requirements for ensuring the identity of parties involved in data exchange FCO_NRO (Non-repudiation of origin), FCO_NRR (Non-repudiation of receipt)
FCS: Cryptographic Support Requirements for cryptographic key management and operations FCS_CKM (Cryptographic key management), FCS_COP (Cryptographic operation)
FDP: User Data Protection Requirements for protecting user data FDP_ACC (Access control policy), FDP_IFC (Information flow control policy)
FIA: Identification and Authentication Requirements for verifying the claimed identity of users FIA_UAU (User authentication), FIA_UID (User identification)
FMT: Security Management Requirements for managing security attributes, data, and functions FMT_MSA (Management of security attributes), FMT_SMR (Security management roles)
